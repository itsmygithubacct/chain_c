# Golden vectors — byte-exact pins from the upstream TypeScript chain layer

> **Note on keys in these vectors.** Any WIF / private key that appears here (e.g.
> `txSighashAddress`) is a **deterministic test vector derived from a fixed constant**
> (`fixed_priv_bn_hex = 0x0101…01`), used only to pin key/address/signature encodings.
> These are public test constants — **not secrets, and they hold no funds**. Do not
> treat a secret scanner hit on this file as an incident.


`golden.json` (pretty-printed) and `broadcast.json` hold byte-exact INPUT +
OUTPUT strings captured by running the **upstream TypeScript implementation** that
`chain_c` is a port of. The C tests pin their outputs against these committed
strings, so any byte-level drift fails loudly. Per-area copies (same `_meta`, one
vector each) live in `by-area/`.

## How these were produced

These vectors are **committed** — you do not need the upstream TypeScript project
or any toolchain to build or test `chain_c`. They were generated by a capture
program in the upstream repo (a `ts-node` script that drives each TypeScript API
under fixed, deterministic fixtures and records inputs + outputs). Re-running it
is deterministic for every vector value (only `_meta.capturedAt` and the
random-salt `commitLimit_live_shape` probe vary between runs); each vector is
wrapped in try/catch and records `{error}` on failure so one bad vector never
loses the rest. `broadcast.json` additionally holds six full, signed,
`tx.verify()`-passing transactions (deploy / executeAction / revoke for both Tea
contracts) that pin the contract **unlocking** scripts.

## Deterministic fixtures

- **Fixed ECDSA priv key**: BN = `0x01`×32 (`_meta.fixed_priv_key_bn_hex`).
  Used for WIF, pubkey, address, tx signing, charter signing, and as the
  PubKey for both elder/agent in the locking-script vectors.
- **Fixed Rabin key**: `Rabin(securityLevel=6).generatePrivKeyFromSeed(0x42×64)`
  (`_meta.rabin_security_level`, `_meta.rabin_seed_hex`). Deterministic primes.

## Notes for the C port

- **Decimal vs hex is load-bearing**: Rabin pubkeys / bigints serialize as
  DECIMAL (`bigint.toString()`); PubKey/Sha256 as lowercase hex. The
  `canonicalJSON` and `ricardianHash` depend on this exactly.
- **`computeProvenanceHash` / `computeRicardianHash` SHA256 the DECODED bytes**
  of the hex preimage, not the ASCII of the hex string.
- `int2ByteString` one-arg is Bitcoin CScriptNum minimal sign-magnitude LE
  (`0`→`""`, `-1`→`81`, `128`→`8000`, `255`→`ff00`). `num2bin`/`int2ByteString(_,w)`
  is fixed-width LE and throws when the value does not fit (recorded as `{error}`).
- The two contract `lockingScript` hexes embed the compiled sCrypt artifacts
  (`artifacts/ricardianTea.json`, `artifacts/src/contracts-next/agentTea.json`)
  with the 13 / 12 constructor args substituted in the fixed order; the C codec
  must reproduce that substitution to match these hexes byte-for-byte.
